# Phase 8 Plan 01: Mapping v2 (Simplificação + Preferências)

## Objective

- **What:** Evoluir `data/mappings.json` para o formato v2 com regras por selector (item/tag) e preferência `prefer_equivalent`, mantendo fallback para v1, e atualizar `startup map` para operar o novo formato via teclado (setas/Enter).
- **Why:** Reduzir o atrito operacional do mapping (menos edição manual e menos pares item↔item), tornando a escolha vanilla vs equivalente previsível e controlável por regra.
- **Output:** Loader v2 + migração v1→v2 em memória, editor v2 com CRUD, e testes cobrindo parsing/migração e lookup por item/tag.

## Mapping v2 Contract (target)

Invariantes para `data/mappings.json` quando `version=2`:

- `version`: número inteiro `2`
- `rules`: array de regras, cada uma com:
  - `selector`: string (`mod:item` para item ou `#namespace:tag/path` para tag)
  - `kind`: `"item"` ou `"tag"` (persistido; deve ser consistente com `selector`)
  - `class`: `"ARMOR_HELM" | "ARMOR_CHEST" | "ARMOR_LEGS" | "ARMOR_BOOTS" | "TOOL_PICK" | "TOOL_SHOVEL" | "TOOL_AXE" | "TOOL_HOE" | "TOOL_SWORD" | "TOOL_BOW" | "TOOL_SHIELD"`
  - `prefer_equivalent`: boolean (opcional; default `false`)
- `tier_overrides`: map `{ ["mod:item"] = "iron|diamond|..." }` (somente para `kind=item` no v2 inicial)
- `gating.by_building_type`: preservado (compatibilidade com progressão já existente)
- Campos extras podem existir (ex.: `notes`) e devem ser ignorados pelo loader

Compatibilidade v1:

- `version=1` e estrutura atual (`items/classes/tier_overrides/gating`) continua carregando.
- Ao ler v1, o loader cria um índice interno compatível (migração em memória) sem reescrever o arquivo.
