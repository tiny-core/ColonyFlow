# Phase 17 Plan 01: Scheduler com Budget (limites por tick)

## Objective

- **What:** Introduzir budgets por tick/janela para requests e chamadas a perifericos (ME/MineColonies/Inventarios) e refatorar o Engine para processar requests em fatias (cursor).
- **Why:** Reduzir picos e travamentos quando ha muitos requests/workorders ou quando o ME/inventarios ficam caros, mantendo corretude via defer/retomada.
- **Output:** Configuracao `[scheduler_budget]`, enforcement centralizado, Engine com cursor, indicador de throttling na UI e testes.

<threat_model>
## Threat Model

### Assets
- Estabilidade do loop (nao travar sob backlog)
- Corretude do fluxo (nao perder requests, nao marcar erro por throttle)
- Liveness (backlog reduz ao longo do tempo, sem starvation)
- Operabilidade (UI/log claros, sem spam)

### Threats (block on: high)
- **T-17-01 (high):** Budget mal configurado quebra operacao (limites muito baixos) -> mitigacao: defaults conservadores; 0/negativo tratado como "sem limite" para opt-out; UI indica throttling com motivo.
- **T-17-02 (high):** Wrappers fazem chamada mesmo quando budget estourou -> mitigacao: budget check imediatamente antes de `Util.safeCall`; erro padrao `budget_exceeded:<kind>`; testes com mock garantindo zero chamadas.
- **T-17-03 (high):** Engine perde progresso/cursor (starvation ou loop infinito) -> mitigacao: cursor monotonic/round-robin; interromper cedo ao estourar budget; testes de avancar cursor.
- **T-17-04 (high):** Throttle vira erro e aumenta `state.stats.errors` ou `waiting_retry` indevido -> mitigacao: tratar `budget_exceeded:*` como "defer"; nao alterar status para erro; logar apenas transicoes.

### Notes
- Qualquer string exibida in-game (UI/print/log) deve ser ASCII-only (sem acentos).
</threat_model>

  </action>

  </acceptance_criteria>

<task id="17-01-04" type="code">
  <name>Refatorar Engine para cursor e requests_per_tick (defer/retomada)</name>
  <files>modules/engine.lua</files>
  <read_first>
    - modules/engine.lua
    - modules/scheduler.lua
    - .planning/phases/17-scheduler-budget-limites-por-tick/17-CONTEXT.md
  </read_first>
  <action>
    - No inicio de `Engine:tick()`:
      - Se existir `state.budget`, chamar `state.budget:beginTick(state)`.
    - Introduzir refresh throttling de requests:
      - Ler `requests_refresh_interval_seconds` de config e manter `self._next_requests_refresh_at_ms`.
      - Atualizar `state.requests` apenas quando `now >= _next_requests_refresh_at_ms`.
    - Processamento por cursor:
      - Ler `requests_per_tick` e processar no maximo esse numero por tick.
      - Manter `self._rq_cursor` (1..#requests) e avancar de forma round-robin.
      - Se qualquer chamada retornar `budget_exceeded:*`, parar o processamento do tick imediatamente, manter cursor e sair sem marcar erro.
    - Corretude:
      - Throttle nao deve alterar `work.status` para erro/retry; apenas adiar.
      - Somente erros reais continuam incrementando `state.stats.errors`.
  </action>
  <acceptance_criteria>
    - Com `requests_per_tick=1` e 3 requests pendentes, o engine avanca o cursor entre ticks (test unit).
    - Quando um wrapper retornar `budget_exceeded:*`, `Engine:tick()` encerra cedo e `state.throttle.active==true`.
  </acceptance_criteria>
</task>
