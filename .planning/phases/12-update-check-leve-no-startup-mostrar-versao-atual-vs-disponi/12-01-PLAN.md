# Phase 12 Plan 01: Update check leve + UI de versao instalada vs disponivel

## Objective

- **What:** Adicionar um update-check leve (manifesto remoto) com cache TTL e exibir versao instalada vs disponivel na UI (dual-monitor) sem interromper a operacao.
- **Why:** Reduz atrito operacional: o operador ve rapidamente se existe update e qual comando rodar, sem depender de terminal/diagnostico manual.
- **Output:** Modulo de update-check + loop em background + ajustes na UI (header + banner no Status) + verificacao.

## Threat Model (ASVS L1)

- **T-12-01 (Alta):** Update-check bloqueia boot (espera rede) ou derruba loops paralelos -> mitigacao: rodar em loop dedicado com `pcall`, retries limitados e nunca executar de forma sincrona no bootstrap.
- **T-12-02 (Media):** Cache persistido corrompido causa erro em runtime e quebra UI -> mitigacao: parse defensivo, defaults seguros, escrita atomica e ignorar cache invalido.
- **T-12-03 (Media):** UI poluida/ilegivel em monitores pequenos (header estoura largura) -> mitigacao: truncamento/abreviacao deterministica e indicador discreto.
</threat_model>

<task type="code">
  <name>Criar modulo de update-check (manifest remoto + cache TTL + status)</name>
  <files>modules/update_check.lua</files>
  <read_first>
    - lib/util.lua
    - lib/version.lua
    - tools/install.lua
    - manifest.json
  </read_first>
  <action>
    Criar `modules/update_check.lua` com responsabilidades claras (sem efeitos colaterais fora de I/O do cache e HTTP):
    - Ler `data/install.json` (quando existir) e montar a URL do manifesto remoto:
      - Defaults devem refletir o instalador (base/ref/manifest_path), mesmo se `data/install.json` nao existir.
    - Implementar cache persistido (ex.: `data/update_check.json`) com campos minimos:
      - `checked_at_ms` (epoch utc ms)
      - `ttl_ms` (padrao 6h)
      - `available_version` (string SemVer) quando conhecido
      - `status` (ex.: ok/no_update, update_available, http_off, error)
      - `err` (string curta, opcional)
      - `manifest_url` (string, opcional)
    - Expor funcao principal para o Scheduler:
      - `UpdateCheck.loadCache()` -> retorna struct normalizada ou `nil`
      - `UpdateCheck.shouldRefresh(cache, nowMs)` -> bool
      - `UpdateCheck.refresh(state, opts)` -> executa fetch remoto com 2 tentativas quando HTTP disponivel:
        - Valida JSON e campo `version` (SemVer simples)
        - Compara com versao instalada (quando existir) via `Version.compare`
        - Em falha remota: manter `available_version` anterior (stale) quando existir e marcar status/err
      - Garantir que todas as strings que vao para UI sejam ASCII.
  </action>
  <acceptance_criteria>
    - Existe `modules/update_check.lua` e ele nao depende de `shell.run`/tools.
    - Cache e parse sao defensivos (nao quebra se arquivo faltar ou JSON for invalido).
    - Existem status claros para: sem HTTP, erro remoto, update disponivel, sem update.
  </acceptance_criteria>
  <verify>
    - `startup test` (apos adicionar testes da fase) passa.
  </verify>
  <done>
    O runtime tem um componente isolado e seguro para realizar update-check e persistir cache.
  </done>

<task type="code">
  <name>Adicionar loop de update-check em background no Scheduler</name>
  <files>modules/scheduler.lua</files>
  <read_first>
    - modules/scheduler.lua
    - modules/update_check.lua
  </read_first>
  <action>
    Em `modules/scheduler.lua`:
    - Adicionar `loopUpdateCheck()` executado em paralelo com engine/ui/eventos.
    - Politica:
      - Nunca bloquear o boot: loop inicia e devolve controle imediatamente.
      - Se TTL expirou: tentar refresh remoto (2 tentativas) e atualizar `state.update`.
      - Se TTL nao expirou: dormir e reavaliar (ou dormir curto).
      - Sempre proteger operacao com `pcall` e backoff em erro.
    - Logs:
      - `INFO` quando update disponivel (incluindo versoes)
      - `INFO` quando HTTP indisponivel/bloqueado ou quando falhar (sem spam: logar apenas quando status muda ou quando tentativa acontece apos expiracao)
  </action>
  <acceptance_criteria>
    - O sistema sobe com UI/engine mesmo quando rede estiver ruim ou HTTP off.
    - Loop nao encerra o programa (nao deve retornar; deve dormir e continuar).
  </acceptance_criteria>
  <verify>
    - Rodar `startup` e observar que a UI aparece imediatamente.
  </verify>
  <done>
    O update-check roda continuamente e de forma resiliente em background.
  </done>
</task>
