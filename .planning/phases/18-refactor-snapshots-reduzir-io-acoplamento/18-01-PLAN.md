# Phase 18 Plan 01: Refactor por Snapshots (reduzir acoplamento/IO)

## Objective

- **What:** Introduzir um snapshot canonico (estrutura unica) produzido pelo Engine e consumido pela UI, reduzindo acoplamento e evitando IO fora do tick do engine.
- **Why:** Facilitar evolucao/otimizacao: UI deixa de depender de detalhes internos e as regras ficam mais testaveis.
- **Output:** `modules/snapshot.lua`, Engine publicando `state.snapshot`, UI usando snapshot, testes e checklist.

<threat_model>
## Threat Model

### Assets
- Estabilidade (nao travar por acoplamento/atualizacoes parciais)
- Corretude (UI representa o estado real do loop)
- Performance (evitar recomputes e IO duplicado)

### Threats (block on: high)
- **T-18-01 (high):** Snapshot inconsistente (UI mostra combinacao de dados de ticks diferentes) -> mitigacao: snapshot construido em um ponto unico do tick e publicado atomicamente (swap de referencia).
- **T-18-02 (high):** UI volta a tocar perifericos (IO escondido) -> mitigacao: API de snapshot cobre tudo que UI precisa; testes detectam chamadas em runtime simulado.
- **T-18-03 (high):** Refactor grande demais quebra fluxo -> mitigacao: mudanca incremental; manter compatibilidade com `state.*` durante transicao.
</threat_model>
