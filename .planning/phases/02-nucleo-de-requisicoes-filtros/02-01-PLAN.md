# Phase 2 Plan 01: Núcleo de Requisições + Filtros

## Objective

- **What:** Implementar normalização de requests, inferência de tiers, resolução de equivalências e reconciliação de destino para calcular faltante real (sem depender de craft/entrega).
- **Why:** Sem um núcleo correto de “o que foi pedido” e “o que falta de verdade”, o sistema crafta demais, entrega errado e fica opaco para operar em modpacks.
- **Output:** Engine refatorada para separar decisão de ação, modelos estáveis de request/work, cache de destino e testes cobrindo tiers/equivalências/faltante.

<task type="code">

  <files>modules/inventory.lua, modules/engine.lua</files>
  <action>
    Introduzir política de leitura de destino que:
    (1) resolve target a partir de delivery.default_target_container (string ou lista) escolhendo o primeiro periférico disponível;
    (2) varre todos os slots e soma quantidades por item;
    (3) usa cache TTL por destino para evitar revarreduras desnecessárias (CACHE-02), com refresh sob demanda por requisição.
    Implementar faltante = max(0, requerido - presente_no_destino) para o item escolhido.
    Se falhar em validar destino (periférico ausente/erro), marcar a work item como waiting_retry com backoff e registrar motivo (DEL-04).
  </action>
  <verify>
    Criar testes com inventário mockado (list()) cobrindo: somatório correto; faltante correto; falha de list() gera waiting_retry; cache reduz leituras repetidas dentro do TTL.
  </verify>
  <done>
    faltante calculado é correto e o sistema não crafta/entrega no escuro quando destino falha.
  </done>
</task>
